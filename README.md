# Retail Inventory Optimization Analytics

An end-to-end retail operations analytics project that uses **SQL, Python, and Streamlit** to improve inventory planning, supplier management, demand forecasting, and reorder decision-making.

The project identifies where stock is overbuilt, where stockout risk is high, which suppliers need attention, which products are driving demand, and what actions the business should take next.

## Live Dashboard

View the deployed Streamlit dashboard:

## Project Summary

Retailers lose money when inventory is not balanced correctly:

- Too much stock increases holding cost, markdown risk, and working-capital lockup.
- Too little stock creates stockouts, lost sales, and poor customer experience.
- Unreliable suppliers make reorder planning less predictable.
- Poor product-store allocation causes some stores to sit on excess stock while others run out.

This project turns raw retail data into an analytics workflow that supports inventory actions such as:

- Reorder
- Transfer stock
- Freeze replenishment
- Monitor supplier reliability
- Prioritize high-demand products
- Reduce carrying cost

## Business Questions Answered

- Which product-store combinations are at restock risk?
- Which SKUs are overstocked or slow-moving?
- Which products and categories generate the most revenue and profit?
- Which stores are performing best?
- Which suppliers are reliable, and which need SLA review?
- What is the 30-day demand forecast by product?
- Which items should the business reorder, transfer, markdown, or monitor?

## Core KPIs

| KPI | Formula / Logic |
|---|---|
| Inventory Value | `current_stock * price_per_unit` |
| Inventory Turnover | Estimated annual product movement compared with current inventory |
| Days of Supply | `current_stock / average_daily_demand` |
| Safety Stock | Demand variability adjusted by lead time |
| Recommended Reorder Point | Lead-time demand + safety stock |
| Recommended Order Quantity | 30-day demand need + safety stock - current stock |
| Holding Cost | `inventory_value * 18%` |
| Supplier On-Time Rate | On-time deliveries / total deliveries |
| Supplier Fill Rate | Quantity received / quantity ordered |
| Supplier Score | Weighted score using on-time rate, fill rate, and delay efficiency |
| 30-Day Forecast | Weighted recent demand forecast |

## Python Analytics Pipeline

The Python pipeline:

1. Loads source CSVs
2. Builds a sales mart
3. Models revenue and gross profit using explicit assumptions
4. Builds product-store inventory optimization KPIs
5. Simulates supplier delivery events where source data is missing
6. Creates a supplier reliability scorecard
7. Forecasts 30-day demand using weighted recent demand
8. Exports dashboard-ready CSV files
9. Writes executive insights

## Business Recommendations

1. **Prioritize restock-risk SKUs**  
   Create purchase orders for product-store rows below recommended reorder point.

2. **Reduce overstock exposure**  
   Freeze replenishment, run markdowns, or transfer stock for high days-of-supply SKUs.

3. **Use supplier scorecards in procurement reviews**  
   Suppliers with low reliability scores should be reviewed for SLA renegotiation or backup sourcing.

4. **Plan around forecasted demand**  
   Use the 30-day demand forecast to prioritize near-term purchasing for high-demand products.

5. **Track holding cost monthly**  
   Inventory value and holding cost should be monitored as working-capital KPIs.

## Data Quality And Assumptions

This project is transparent about assumptions:

- The data is synthetic.
- Supplier delivery history is not present, so supplier delivery events are simulated.
- Source sales amounts are inconsistent with units and prices, so modeled revenue is calculated from units and unit price.
- Cost of goods sold is estimated at `62%` of listed unit price.
- Holding cost is estimated at `18%` of inventory value.
- Forecasting is a decision-support layer, not a production-grade demand planning system.
